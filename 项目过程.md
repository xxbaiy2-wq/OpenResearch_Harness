



## 理解

ORM = Object-Relational Mapping（对象关系映射）
让你用 Python 代码操作数据库，而不用写 SQL。

什么是 SQLAlchemy 的 Engine 和 Session？
维持和 MySQL 的连接，管理一堆 TCP 连接
一次完整的"交流"：你可能问了好几个问题
Session.close()

什么是 FastAPI 的依赖注入（Dependency Injection）？
你不用手动创建/关闭 db，FastAPI 帮你管理。这就是依赖注入。

### core\

- 写了backend\.env，注意DATABASE_URL 的格式：
数据库类型+驱动://用户名:密码@主机:端口/数据库名
     ↓       ↓       ↓   ↓   ↓    ↓
  mysql+pymysql  root  pw localhost 3306  openresearch
其中驱动，pymysql：你的 Python 代码  →  pymysql（翻译官）  →  MySQL 数据库

Python 找模块是相对于你运行命令的目录的。你从 backend/ 运行，Python 在 backend/ 下找 core/，但 core/ 实际在 backend/app/core/；
- __init__.py
Python 把文件夹当成"可以导入的包"，需要文件夹里有一个 __init__.py 文件。没有它，Python 就不认这个文件夹是包。

### models\
用 ORM 定义数据库表的结构。
research_tasks（任务）  ──1:N──>  agent_runs（每次执行）
                                    │
                          ┌─────────┼──────────┐
                          ↓         ↓          ↓
                   agent_steps   tool_calls   verification_results
                   (每个步骤)   (每次工具调用)   (验证结果)
                                                            ↓
                                                   daily_reports（最终报告）

research_tasks ──1:N──> research_items（收集到的资料）


### schemas\
Schema = 数据的"表单模板"，定义 API 请求或响应应该长什么样。你填的存款单
在 ORM 语境下，Model = 数据库表的"Python 化身"。银行内部系统里的账户表

真实的 MySQL 数据库
    ↑  ORM 映射
  Model（ResearchTask）  ← 这是数据库的"Python 翻译"
    ↑  API 转换
  Schema（TaskCreate / TaskOut）  ← 这是给 API 用的"表单模板"
    ↑  HTTP 请求/响应
  用户的浏览器/前端


Pydantic 模型
schemas 和 models 有什么不同？（八股考点）
场景：用户创建一个研究任务
数据库里（models）的 ResearchTask 长这样：

id	topic	status	created_at	updated_at
1	"Agent Harness"	"pending"	2026-05-31 10:00	2026-05-31 10:00
用户发来的请求（schema）长这样：


{
  "topic": "Agent Harness",
  "description": "追踪 Agent 相关动态"
}
注意区别：

Model (数据库)	Schema (API)
id	有，数据库自增	没有，用户不需要传
status	有，有默认值	没有，创建时不该由用户设置
topic	有	有
created_at	有，服务器自动填	没有
一句话总结：

Model = 数据库认为一张表应该长什么样
Schema = API 接口认为请求/响应应该长什么样
两者字段不完全一样，所以不能用同一个类

POST /api/tasks         → 请求需要 TaskCreate, 响应需要 TaskOut
GET  /api/tasks         → 响应需要 TaskOut 列表
POST /api/tasks/1/run   → 响应需要 RunOut
GET  /api/agent-runs/1  → 响应需要 RunDetail（包含 steps + tool_calls + verification）

## 核心部分
User 点击 Run
    ↓
Runner（指挥官）：好，开始执行！按步骤来
    ↓
步骤1: Planner    → 计划要搜什么
步骤2: Search     → 调用工具搜集资料（tool_registry 找工具 → permission_guard 检查 → 执行工具）
步骤3: Fetch      → 获取详细内容
步骤4: Rank       → 排序
步骤5: Summarize  → 生成报告草稿
步骤6: Verify     → 检查质量
步骤7: Publish    → 发布报告
    ↓
每一步都经过 Trace Logger 记录到 MySQL
